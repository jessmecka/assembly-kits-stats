Methodology (updated): "Approved" now counts individual good-verdict build cycles (one full kit build = one reviewed segmentReviews entry, ≥60s, from the latest review session per recording) — not whole recordings. A single ~45-min recording typically contains 8–11 cycles, so these numbers are much higher than a recording count. Cycles are only counted from recordings whose overall review status is approved (good builds inside rejected recordings are excluded — contested pool, not yet spot-checked). "Rejected" and "Rej %" are recording-level QA outcomes. "Pending" is unreviewed recordings awaiting review (no cycle data yet). Requirement (updated): each task must deliver 100 approved cycles or 20 hours of approved cycle time, whichever is more — a task is complete only when it has ≥100 cycles and ≥20h of cycle time. Fast-cycle kits (short builds) therefore need more than 100 cycles to reach the 20h floor. Source: qa_usertask_review_sessions.segmentReviews + user_tasks, mecka-ai MongoDB (Longhua + Toronto).
